## 1. 기준 시스템 해부도 — MiroFish

> MiroFish: 오픈소스(AGPL) 멀티에이전트 시뮬레이션 엔진. 시드 문서(뉴스/정책/소설)에서
> 지식그래프를 구축하고, 독립 메모리를 가진 수백~수천 페르소나 에이전트가 소셜 환경에서
> 상호작용해 여론·서사 전개를 예측. 학부생 1인이 AI 코딩 어시스턴트로 ~10일 구축.
> 로컬 스택 포크 존재(Neo4j + Ollama).

각 행 3필드: [챕터 매핑] [도구 후보] [현재 상태: 구축가능 / 부분 / 모름]

| # | 컴포넌트 | 역할 | 챕터 매핑 | 도구 후보 | 현재 상태 |
|---|---|---|---|---|---|
| C1 | 시드 처리 | 입력 문서 → 엔티티/관계 추출 | Ch.03(구조화 출력), Ch.05(청킹) | LLM + JSON Schema 강제 | 부분 |
| C2 | 지식그래프 구축 (GraphRAG) | 엔티티·관계를 명시적 그래프로 | Ch.05(RAG 보강: 인덱싱·멀티홉), Ch.01(그래프) | Neo4j / NetworkX | 모름 |
| C3 | 페르소나 생성 | 성격·편향·영향력·반응속도를 가진 에이전트 대량 생성 | Ch.08(Agents/Rules), Ch.03(템플릿) | 프롬프트 템플릿 + 파라미터 샘플링 | 부분 |
| C4 | 에이전트별 독립 메모리 | 개별 대화 이력·의견 변화 지속 | Ch.09(episodic/계층/망각) | Zep 대체: 로컬 벡터DB or 파일 기반 | 모름 |
| C5 | 시뮬레이션 실행 | 이중 플랫폼(마이크로블로그/포럼)에서 상호작용, 스케줄링 | Ch.08(활성화·스케줄링·ODD), Ch.07(토폴로지), Ch.10(상태) | OASIS 또는 자체 이벤트 루프 | 모름 |
| C6 | ReportAgent (관측·분석) | 에이전트를 감시하는 에이전트, 집단 동학 해석 | Ch.11(평가), Ch.16(관측 UI) | 별도 분석 에이전트 + 지표 집계 | 부분 |
| C7 | 계측·재현성 | 시드 고정, 실행 로그, 반복실험 | Ch.08(random seed & replication), Ch.15(트레이싱) | 로그 + correlation ID | 부분 |
| C8 | 배포·운영 | docker-compose, 프론트/백엔드 | Ch.14(서빙) | Docker | 구축가능 |

**해석 규칙**: "모름" 컴포넌트가 곧 v0.3~0.6 실험의 원료다.
L2 게이트 = 전 컴포넌트가 "구축가능"으로 전환 + 미니 MiroFish 완주.

**미니 MiroFish 정의 (L2 게이트용)**: 시드 문서 → 그래프 → 페르소나 ≥50 →
시뮬레이션 → 리포트, 전 파이프라인 1커맨드. Xenkai 실작업(독자 반응 예측) 1회 사용.

## 2. 17챕터 3계층 절단

### 계층 A — 실험 척추 (버전 가설로 직접 소환)

| 챕터 | 핵심 | 소환 버전 |
|---|---|---|
| Ch.03 Prompt·Program | 구조화 출력, 검증 루프 | v0.1~ 상시 |
| Ch.04 Tool·Function calling | 스키마 계약, MCP, 게이팅, 멱등성 | v0.3 |
| Ch.06 Agent 아키텍처 | 실행 패턴, 예산, 정지 조건 | v0.2~0.4 |
| Ch.07 Multi-Agent | 토폴로지, 프로토콜, 합의, 루프 감지, 지표 | v0.5 |
| Ch.08 ABM | 네트워크, 스케줄링, ODD, 스윕 | L2 게이트, v0.9 |
| Ch.09 Memory | 계층, 망각, 오염 방어, 평가지표 | v0.6 |
| Ch.10 상태·실행 | 상태기계, 재시도, 보상, 재현 | v0.4 |
| Ch.11 Evaluation | 하니스, ablation, 멀티에이전트 지표 | v0.7 (전 버전 기저) |
| Ch.12 검증·테스트 | 계약 테스트, 카오스, 리플레이 | v0.3, v0.7 |

### 계층 B — 증상 소환 (해당 증상 발생 시에만)

| 챕터 | 소환 트리거 (증상) |
|---|---|
| Ch.13 Safety·Guardrails | 툴 오남용/메모리 주입 의심, 고위험 툴 도입 시 |
| Ch.14 LLMOps·Serving | 지연/비용이 채점표 B에서 병목으로 판정될 때 |
| Ch.15 분산 시스템 | 멀티에이전트에서 교착/순서 역전/부분 실패 발생 시 (v0.5 예상) |
| Ch.16 Human-AI Interface | 개입 횟수가 정체되어 관측 UI가 병목일 때 |
| Ch.17 거버넌스 | 제2 숙주 이후 자산/권한 관리가 병목일 때 |

> 원리: 증상 주도 학습이 순서 주도 학습보다 착용률이 높다.
> 부검에서 증상이 특정되면 그 부검 안에서 해당 챕터를 흡수한다.
